# Rewrite Roadmap

Arcana is being rewritten under a hard language freeze until selfhost.
This roadmap tracks the current rewrite slice, not the full post-selfhost program.

- The previous higher-level grimoire experiment has been removed. Any future desktop, graphics, text, or higher-level audio layer now needs to be reintroduced on top of the retained `grimoires/arcana/*` substrate with fresh scope approval rather than by inheriting deleted package boundaries.
- The later owned-showcase proof remains future work: current substrate demos and scaffold grimoire checks prove backend/runtime and package seams, but they do not yet count as a finished showcase app built through completed Arcana-owned grimoires.

Imported-source guardrails:
- archived MeadowLang app/toolchain corpus was a bootstrap hotspot import and must not be reintroduced as an implementation template
- `std` itself is rewrite-owned first-party library surface; historical imports only provided transition context
- future higher-level non-std package responsibilities are frozen by role, not by carried Meadow-era package names
- rewrite-owned core grimoires live under `grimoires/arcana/*`; the broad historical reference tree is archived outside the repo
- any future higher-level non-std packages will need to be reintroduced deliberately rather than inferred from deleted tree structure
- imported standard-library contents do not automatically define correct standard-library layering; showcase/game/demo logic that leaked into `std` should move back into showcase or app grimoires unless it is explicitly ratified as general-purpose std surface
- first-party host/io/window/input/canvas/time/audio plus primitive graphics/text remain required before selfhost so the rewrite can show real apps/showcases and bootstrap future Arcana-owned grimoires, but the old MeadowLang VM/bytecode/window stack is not the mandated implementation path
- those host/app/runtime packages are real Rust rewrite commitments, not a promise that imported `std` or carried grimoires will keep satisfying them unchanged
- third-party Rust crates may still appear underneath the runtime/backend, but only as replaceable private implementation details rather than the defining first-party substrate
- parser opaque-expression fallback and runtime/resource handles are different concerns: the former should be eliminated from the frontend path before selfhost, while current source-declared opaque handle families such as desktop windows/frame handles, retired image handles, file streams, and `Audio*` remain bootstrap seams and must be revisited once the rewrite-owned runtime/backend resource model is settled
- ECS scheduling/components remain first-party language/runtime surface during the rewrite; they are not merely showcase helpers carried from MeadowLang
- carried `std.app` fixed-step helpers and `std.tooling` planner helpers are convenience corpus, not rewrite-owned architecture until a scope explicitly ratifies them
- carried generated snapshot artifacts such as direct-emit spec shards are not rewrite architecture evidence by themselves; use them to track migration work, not to override approved contracts or crate-side implementation status
- historical MeadowLang reference corpus is now archived outside the repo and no longer participates in normal default validation.

## Non-Goals Before Selfhost

- no language expansion
- no remote registry or Git dependencies
- no public bytecode compatibility promise
