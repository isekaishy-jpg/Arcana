Actionable synthesis:

The 55 findings do not represent 55 independent jobs. Most of them collapse into a smaller set of root causes.

Superseding work items:

1. Executable backend contract closure.
   This is the highest leverage implementation item. The runtime is still reconstructing too much source behavior from bootstrap artifacts and heuristics instead of executing a richer lowered contract. This one cluster covers findings 3, 4, 5, 9, 12, 13, 14, 15, 17, 39, 40, 41, 42, 43, 44, 45, and 48.
   What it really means:
   - stop relying on runtime-side qualifier/method/path guessing as the source of truth
   - carry resolved executable identity and operation kind through frontend/HIR/IR/AOT
   - make the runtime executor understand the actual lowered expression/statement families Arcana already parses
   Why it supersedes so much:
   - removes many executor-owned std shims
   - fixes dotted qualifiers, `?`, `>>`, and memory phrase constructor behavior on the right layer
   - gives a natural place to close `Index` / `Slice` / `RangeInt` / `for` execution and chain-expression execution
   - reduces the chance of more bootstrap-only gaps surfacing later

2. Real scheduler/worker/task substrate.
   This is the next broad implementation item and probably the single largest feature gap inherited from the Meadow comparison. It covers findings 2, 6, 7, 20, 27, 30, 34, 35, and 36.
   What it really means:
   - replace eager done-handle `weave` / `split` behavior with a real execution substrate
   - settle how async entry, task graphs, worker affinity, and chain parallel/async lowering actually execute
   - make `thread_id` and main-thread-only rules meaningful
   Why it supersedes so much:
   - it resolves most concurrency findings in one architectural move
   - it is the real enabling primitive for multithreaded chain execution
   - it is also where `async fn main` stops being a special-case gap and becomes a normal entry mode

3. Structured type-law model for `where`.
   This is the broadest remaining language-law gap. It covers findings 49, 51, and 52 directly, and it is tightly related to 25.
   What it really means:
   - stop treating `where` as mostly opaque surface text
   - add structured predicate carriage for trait bounds, projection equality, and outlives predicates
   - decide what is actually frozen and validate it semantically
   Why it supersedes so much:
   - projection equality, supertrait-style bounds, and outlives law are all symptoms of the same missing intermediate representation and semantic pass
   - this is the type-system analogue of the executable-backend-contract problem

4. Ownership and resource-law extraction.
   This is the main contract-side root item for `take` and handle/resource semantics. It covers findings 1, 19, and 24 directly, and it influences 37 and 38.
   What it really means:
   - ratify access-mode law beyond the current top-level prose
   - define consuming-call equivalence for user routines, std wrappers, and host intrinsics
   - clarify opaque-handle/resource lifecycle rules enough that runtime work is not guessing
   Why it supersedes so much:
   - a cleaner ownership contract will simplify both runtime implementation and std surface expectations
   - without it, every handle/resource feature keeps re-opening the same `take` questions

5. Contract and authority cleanup.
   This is mostly doc/spec hygiene, but it is still necessary because some findings are caused by broken or stale authority rather than missing code. It covers findings 18, 21, 22, 23, 25, 26, and 27.
   What it really means:
   - repair broken selfhost matrix/doc links
   - mark stale Meadow-era docs as non-authoritative where needed
   - extract missing rewrite-era domain specs from `docs/arcana-v0.md`
   Why it matters:
   - prevents repeated false pressure during implementation and review
   - makes later runtime/backend work less likely to target the wrong contract

6. Intentional narrowing versus real parity gap review.
   This is a smaller language-policy track, not a first implementation target. It covers findings 22, 28, 46, 47, 50, 53, 54, and 55.
   What it really means:
   - separate "should return in redesigned form" from "should remain intentionally narrowed" from "rewrite gain"
   - examples: tuple patterns, empty `[]`, map literals, guards, `const`
   Why it matters:
   - avoids wasting implementation effort restoring Meadow behavior that Arcana should not simply copy back
